# Mutinynet end-to-end run: LND API privacy-surface map

A single throwaway run on 2026-05-02 funded a Voltage-hosted LND node (Mutinynet/signet) from the public faucet, sent some sats on-chain, opened a Lightning channel to the faucet's LN node, paid an invoice, and cooperatively closed. Below is the per-call accounting of *what each API surface revealed* — the input doc for designing the AI ↔ bitcoind/LND privacy proxy.

## Setup snapshot
- Node A: `first-test.u.voltageapp.io` (LND v0.20.0-beta, Voltage-hosted)
- Counterparty: Mutinynet faucet LND node `02465ed5...@45.79.52.207:9735`
- Chain access (for our own polling): `https://mutinynet.com/api` (public esplora)
- All ledger txids in `~/spacer/test-harness/state/{onchain_send,channel_open,channel_close}.json`

## Per-call leak accounting

### `lncli newaddress p2wkh`
**Returns**: a new bech32 address.
**External exposure**: none yet — but once handed out (to faucet, counterparty, etc.) it's permanently linkable to this node.
**Proxy filter**: AI sees a *funding-pool token*, not the address. Proxy maintains the address↔pool mapping internally.

### `POST faucet/api/onchain {address, sats}` (third-party HTTP)
**External exposure**: faucet sees our address + amount; if auth is GitHub OAuth, address is linked to a GitHub identity.
**Inherent leak**: any third-party funding source learns our receiving address. Mitigate by funding from sources we already trust, or by routing funding through an intermediate wallet.

### `lncli walletbalance`
**Returns**: total / confirmed / unconfirmed / locked / `reserved_balance_anchor_chan` / per-account.
**Proxy filter**: AI usually needs *capability* not exact integers. Return banded values ("≥ 50k", "between 100k and 1M") or boolean predicates ("can fund a 50k channel: yes").

### `lncli listunspent`
**Returns**: per-UTXO address, amount_sat, pk_script, outpoint, confirmations.
**Highest local-leak call.** Full UTXO set = complete coin-control fingerprint, reveals address-clustering and change linkages.
**Proxy filter**: AI must never see raw UTXOs. Proxy answers higher-level questions ("is there a ≥X confirmed UTXO available") without exposing the set.

### `lncli sendcoins --addr=A --amt=N`
**Returns**: txid.
**Inherent leak**: tx broadcast IS publication. Inputs (our UTXOs), outputs (recipient + change) all on-chain forever.
**Proxy filter**: validate destination against allowlist before broadcasting. Coin selection happens inside proxy without exposing UTXO choice to AI.

### `lncli connect <pubkey@host:port>`
**Returns**: status.
**External exposure**: peer learns our pubkey (already public if we have public channels) and our outbound IP.
**Proxy filter**: connect target must come from allowlist. Otherwise the connect surface is a generic "make this node TCP-connect to arbitrary IP" primitive — useful as a side-channel for whatever's calling it.

### `lncli listpeers`
**Returns**: per-peer pub_key, address, feature bitmask, traffic counters.
**Proxy filter**: default to count + grouping; per-peer pubkey/IP only on explicit need.

### `lncli openchannel --node_key=PUB --local_amt=N`
**Returns**: funding_txid.
**External exposure**: funding tx + outputs immediately on-chain. After confirmations, public channel gossiped: pubkey-pair + capacity + scid go to the entire LN graph.
**Proxy filter**: amount + counterparty should be policy-driven. Default `--private` (un-announced) when announcement isn't required. Round/band channel sizes (10k/50k/200k tiers) so AI doesn't see exact funding sat amounts.

### `lncli pendingchannels`
**Returns**: per-pending channel: remote_node_pub, capacity, confirmation_height.
**Proxy filter**: AI usually needs only "any pending? yes/no", not detail.

### `lncli listchannels`
**Returns** per channel: `remote_pubkey, channel_point, chan_id, scid, capacity, local_balance, remote_balance, commit_fee, total_satoshis_sent, total_satoshis_received, num_updates, peer_alias`, all constraints, status flags, commitment_type.
**Highest cross-cutting leak.** Combined with listpeers, this is the full picture of LN topology + balances + activity history.
**Proxy filter**: default = counts + aggregate balances. Per-channel detail requires explicit justification per call. Strip `total_satoshis_*` / `num_updates` (activity history) by default.

### `POST faucet/api/bolt11 {amount_sats}` (third-party HTTP, no auth)
**Returns**: bolt11 invoice — encodes destination pubkey, amount, payment_hash, payment_addr, route_hints, features.
**Inherent leak**: bolt11 spec requires destination pubkey + amount in the encoded blob. Anyone with the invoice has those.
**Proxy implication**: when AI hands proxy an invoice to pay, proxy decodes it internally and exposes only policy-relevant facts ("destination matches allowlist: yes", "amount ≤ ceiling: yes"). Raw invoice is held by proxy.

### `lncli decodepayreq <bolt11>` (local-only)
**Returns**: destination, payment_hash, num_satoshis, expiry, cltv_expiry, route_hints, features, blinded_paths.
**Use this in the proxy** to introspect invoices without giving the AI raw bolt11 strings.

### `lncli payinvoice --force <bolt11>`
**Returns**: per-attempt HTLC table (state, attempt_time, resolve_time, receiver_amt, fee, timelock, **chan_out**, route alias chain), final **preimage** on success.
**Local-leak fields to filter**:
  - `chan_out` — which channel we routed through (reveals topology choice)
  - `route` — full hop sequence
  - `preimage` — proves payment was settled to anyone holding the payment_hash; sensitive
**Proxy filter**: default surface = `{succeeded: bool, paid_amount: banded, fee: banded}`. Route, channel selection, preimage held inside proxy.

### `lncli channelbalance`
**Returns**: aggregate balance + buckets.
**Proxy filter**: same banding treatment as walletbalance.

### `lncli closechannel --funding_txid=T --output_index=N`
**Returns**: closing_txid, then status updates.
**External exposure**: closing tx outputs publish the final balance split between parties at close moment.
**Proxy filter**: which channel to close = policy decision, not AI choice. Force-close especially needs policy guards (force-close is an irreversible signal of distrust).

## Side-channels worth tracking
- **Hosting operator (Voltage)** sees everything LND does regardless of our proxy. The privacy boundary the user wants is AI ↔ LND, not LND ↔ infra. Self-host if operator-trust matters.
- **Public esplora** (mutinynet.com/api) we used to poll tx confirmations: each `/api/tx/<txid>` lookup links our IP to that txid in the operator's logs. Production should use a local esplora or query bitcoind directly.
- **Faucet GitHub auth**: ties our funding to a GitHub identity. For real privacy, swap funding sources or self-host one.

## Concrete recommendations for the proxy
1. **Tokenize identifiers** — addresses, pubkeys, channel_ids, payment_hashes — AI sees opaque tokens; proxy maps internally.
2. **Band numeric values** — balances, fees, amounts as ranges/buckets unless precision is justified.
3. **Allowlist outbound** — every network-touching call (connect, sendcoins, openchannel, payinvoice) checks a policy.
4. **Aggregate-by-default** — list-style calls return counts/summaries; per-item detail requires justification.
5. **Withhold proofs** — preimages, signatures, macaroons stay inside the proxy.
6. **Default `--private` channels** unless announcement is needed.
7. **Audit log** every AI request + proxy decision (the AI may be malicious or compromised).
8. **Side-channel hygiene**: route chain queries through local esplora/bitcoind, not public ones.
