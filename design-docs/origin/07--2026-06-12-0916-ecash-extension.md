**Date:** 2026-06-12 (reconciled 2026-06-12, post-build)

## 1. Scope

Two adversary surfaces, deliberately in one doc because the extension is defined by their interaction: the **AI client** (the standing adversary; see [AI-facing privacy](../../GLOSSARY.md#ai-facing-privacy)) and the **[mint](../../GLOSSARY.md#mint)** - a new, *external* third party. The mint is NOT arbiter infrastructure: the no-internal-mitigations rule (doc 03 §1: the arbiter <-> local bitcoind / LND link carries no local anti-timing mitigation) does **not** apply to it. Mint-facing mitigations are in scope. World-facing concerns (Tor to the mint, LN routing observers, chain observers) remain out of scope per the origin README rule; where a leak is visible to both the mint and the world, the cell says so.

The boundary picture gains one edge. Today: AI <-> [privacy gateway](../../GLOSSARY.md#privacy-gateway) <-> daemons. With the extension: the AI's environment ALSO talks HTTPS directly to the mint for token operations. That second edge is the design's point (autonomy) and its principal new cost (§5.2): **deploying ecash mode sanctions a direct external network channel from the AI environment.** Operators for whom that is unacceptable must not enable the extension.

## 2. Protocol choice (confirmed, not assumed)

**Cashu, single mint, via the nutshell reference implementation.** The working assumption from the epic holds up:

- **Cashu** (NUT specs): blind-signature eCash over Lightning. BDHKE blinding gives the one property a bearer float in AI custody needs - the mint cannot cryptographically link token issuance to later redemption, so the AI's direct mint traffic does not automatically deanonymize the funding source. Funding and defunding are LN-native (bolt11 mint / melt quotes), which is exactly the rail beneath us. Standardized token serialization (V4 `cashuB`, CBOR) gives the handoff format for free.
- **nutshell**: the reference Python implementation. Supplies a CLI wallet the arbiter can shell out to (mirroring the `bitcoin-cli` / `lncli` subprocess pattern, doc 05 §4.2-4.3), and a real mint for the live test environment (sp-2hwco4.4). The arbiter's own code stays minimal per doc 05 §2.1; nutshell sits in `bin/`-style upstream territory like `lncli` does. **Build status (verified sp-uwa0v0):** the wrapper's nutshell subcommand surface (`invoice --no-check` / `invoice --id` for quote/issue, `send -y -d`, `receive`, `pay -y`, `decode`, `balance`, `info`) is verified against nutshell 0.18.1 at the live mint (findings companion §3); every call returns raw stdout text (nutshell prints human-oriented text, not JSON) except `decode` (JSON), and the executor parses it. Corrections the live run forced: `pay`/`send` need `-y` (no operator at stdin), `send` needs `-d` (DLEQ), and `--wallet NAME` must be avoided - it breaks `receive` (doc 08 §4).
- **Fedimint, considered and rejected for this phase:** federated custody would distribute mint risk, but the operational footprint (federation setup, guardians, consensus) conflicts with the §2.1 minimalism discipline, and custody risk is already bounded by the allowance (§8). Revisit only if float sizes ever outgrow the petty-cash framing.
- **Single mint per deployment, operator-chosen, pinned in arbiter config.** Multi-mint would spread custody risk but multiplies the correlating adversaries, forces every timing mitigation to hold at N mints, and grows arbiter code. The allowance already caps what a mint failure can cost. The mint URL is **never AI-suppliable** - an AI-chosen "mint" is a colluding counterparty that converts the float into an exfiltration channel. As built, the pin is the `CASHU_MINT_URL` environment variable with **no default**: an unconfigured deployment raises before any subprocess runs rather than falling back to some well-known public mint. The gate does not parse tokens, so a token minted elsewhere is refused at defund *execution*, not at the gate - the arbiter wallet only talks to the pinned mint, and only the wallet can decode a token's embedded mint URL, so the swap-claim is where the rule bites (a build clarification of the original "refused at the defund gate" wording).
- **Mint selection guidance:** a busy, public, operator-unaffiliated mint. Self-hosting inverts both properties we need: the URL (visible to the AI inside every token) would identify the operator, and the anonymity set at the mint would collapse to our own traffic (§6). A busy public mint makes its other users our crowd.
- **DLEQ verification (NUT-12) is mandatory** in both wallets (arbiter and petitioner). Without it a malicious mint can sign with a per-client key and tag tokens, voiding the blinding. This is the eCash analogue of doc 01's pathfinding-score lockdown: a quiet capability that breaks unlinkability if left on default-trust.

## 3. Custody split

The organizing rule: **every custody crossing is gateway-mediated; movement within the AI's float is not.** Funding and defunding move value between the operator's wallet (LN liquidity) and the AI's float - exactly the crossings the [registry](../../GLOSSARY.md#recipient-address-registry) / [standing approvals](../../GLOSSARY.md#standing-approvals) / [HITL](../../GLOSSARY.md#human-in-the-loop-hitl-approval) / [timing layer](../../GLOSSARY.md#action-delay) stack exists to control. Token operations move value already inside the float, which is already at risk by construction; mediating them would only re-create the latency the extension exists to remove.

| Operation | Path | Gates |
|---|---|---|
| `fund_ecash` (operator wallet -> float) | gateway, full write pipeline | mode gate -> allowance cap -> standing approvals / HITL -> action delay -> execute -> result delay |
| `defund_ecash` (float -> operator wallet) | gateway, full write pipeline | mode gate -> standing approvals / HITL -> action delay -> execute -> result delay |
| hold / count float | AI-direct, local | none (bearer custody) |
| send tokens to a third party | AI-direct (offline-capable string handoff; optional pre-swap at mint) | none - the autonomy the extension buys |
| receive tokens (swap to claim) | AI-direct at the mint | none |
| check proof state | AI-direct at the mint | none |

**Fund flow.** `petcli advanced ecash fund --amount-sats N` -> op `fund_ecash`. Gateway: ecash-mode gate; allowance check (§8); standing-approvals match on (op, amount band) - the destination is structurally fixed (the configured mint), so no `recipient_token` is involved and rules name the literal destination `mint` (or `any`); no match parks in HITL. An amount the gate cannot bound (missing, non-integer, non-positive) fails the allowance check outright, mirroring how an unknown amount fails any bounded approval rule. After the [action delay](../../GLOSSARY.md#action-delay) (lightning-rail window, §7) the arbiter executes: request a mint quote for N sats, pay the quote's bolt11 via `lnd.py`, confirm the quote paid, submit blinded outputs, receive proofs - with randomized gaps between the steps (§6, T1). The serialized token is deposited in the [result registry](../../GLOSSARY.md#result-delay) after the result delay; the AI polls, receives the token, and swaps it into its own wallet (its first AI-direct act).

**Defund flow.** `petcli advanced ecash defund --token <cashuB...>` -> op `defund_ecash` with the token in the request body. Same pipeline minus the allowance check (defund only shrinks exposure). Defund carries no gate-time amount - the gate does not decode the token - so only an unbounded approval rule (no `max_amount_sats`) matches it, which is the right shape for an op that reduces exposure regardless of size. At execution the arbiter swap-claims the token at the mint, then melts to a fresh invoice from its own LND node. **Double-spend race, resolved by the allowance framing:** the AI could spend the same proofs during the action-delay window, making the arbiter's claim fail - but that value was already AI float, so the operator loses nothing; the result reports failure and the [outstanding ledger](#8-allowance-and-blast-radius) is unchanged. No special fast-claim carve-out is needed; defund waits out the standard window like every other write.

**Token handoff format:** the standard V4 `cashuB` serialized token, riding the existing protocol channels (result-registry payload outbound, request field inbound). It never touches the console, so the HITL no-clipboard / hand-retype rule (doc 05 §6) is unaffected - that rule governs console-issued handles like 6-char registry tokens, and a several-hundred-char CBOR token never needs retyping. Build outcome (sp-2hwco4.2): the gateway's existing 64 KiB `MAX_REQUEST_BYTES` already accommodates a petty-cash float's defund token (a few hundred bytes to a few KiB); no cap change was needed.

**Where the wallets live.** The arbiter's transient eCash custody (during fund/defund execution) uses a nutshell CLI wallet under a gitignored `arbiter/ecash/` (mirroring `lnd/`; landed as the fifth gitignored runtime subtree in both `.gitignore` copies, doc 06 §3), shelled to via argv-list subprocess, lazily imported (§9). As built, one `_run` helper pins the whole connection surface: binary (`CASHU_BIN`), mint (`CASHU_MINT_URL`, §2), wallet name (`CASHU_WALLET`), and data dir (`CASHU_DIR`, set explicitly on the subprocess env so an inherited value cannot silently redirect the wallet), with a 60 s default timeout (`CASHU_TIMEOUT_S`) - mint calls are HTTPS round-trips to an external host and a melt waits on an LN payment the mint makes, unlike the local-IPC daemons. The AI's float wallet lives **petitioner-side, in the AI's environment, by design** - the petitioner "holds no secrets" rule (doc 05 §2) is preserved under its real meaning: no *operator* secrets. The float is the AI's own bearer instrument; pretending to custody it server-side would re-mediate what the extension exists to un-mediate.

## 4. Threat model

| Adversary | Observes | Concern |
|---|---|---|
| **The AI client** | every gateway response; its own float exactly (it holds the proofs); the mint URL, keysets, denominations; everything its direct mint channel returns | malicious by design, compromised, or prompt-injected; may exfiltrate value through the float, exfiltrate data over the mint channel, or probe the funding gates |
| **The mint** (new, external) | every mint API call: amounts, keyset epochs, timing, caller IPs; the LN legs of mint/melt quotes; spent-proof secrets at swap | correlate eCash operations with the LN transactions that fund or redeem them; profile the operator's float lifecycle; tag clients (defeated by NUT-12 DLEQ checks); rug the float (custody) |
| **Network observer** (AI <-> gateway link; AI <-> mint link) | traffic metadata even under TLS | timing, connection-count, size fingerprinting; the AI <-> mint link newly exists and is observable |

**What makes the mint distinct from the daemons:** bitcoind and LND are operator infrastructure inside the trust boundary; the mint is a counterparty. It holds the float's backing funds (custodial), it serves the AI directly (so it observes the AI's behavior, not just the arbiter's), and it is the one party positioned to bridge the two surfaces - linking the arbiter's LN identity (which funds and redeems) to the AI client's identity (which swaps and spends). The blinding removes the cryptographic bridge; §6 closes the side channels (timing, amounts) that remain.

Out of scope: the mint host's OS-level operator (sees everything the mint sees; same trust decision); LN route/gossip observers and chain observers (world-facing, archived doc); OS or hypervisor compromise; physical access.

## 5. Per-op leak surface

Severity scale as in docs 01/03: **HIGH** = identifier or full-value reveal; **MED** = amounts, counts, linkable patterns; **LOW** = flags or public data.

### 5.1 What the mint sees, per op

| Op (NUT endpoint) | Caller | Mint observes | Sev | Mitigation |
|---|---|---|---|---|
| mint quote (`/v1/mint/quote/bolt11`) | arbiter | requested amount, caller IP, timing; issues the funding invoice | MED | randomized non-round amounts (§6 T3); execution inside the action-delay window; Tor is world-facing defense-in-depth |
| LN payment of the quote invoice | arbiter's LND | HTLC arrival, amount, timing; payer identity shielded by onion routing but exposed if we share a channel or first hop with the mint ([first-hop attribution](../../GLOSSARY.md#first-hop-attribution)) | MED | no direct channel to the mint's node; route diversity is world-facing |
| mint / issuance (`/v1/mint/bolt11`) | arbiter | blinded outputs (denominations visible), quote id linking issuance to the paid invoice | MED | blinding makes *future redemption* unlinkable; the issuance-to-funding link is inherent and accepted - what must not link is issuance to the AI's later swaps (§6 T2) |
| swap (`/v1/swap`) | AI (receive, reorganize) and arbiter (defund claim) | spent input proofs, new blinded outputs, amounts, keyset epoch, caller IP, timing | **HIGH** (correlation potential) | the blinding breaks proof-level linkage to issuance; residual amount + timing channels are §6's subject |
| melt quote + melt (`/v1/melt/quote/bolt11`, `/v1/melt/bolt11`) | arbiter (defund) | proofs in, **and the bolt11 to pay: our LND node pubkey, amount** ([BOLT11 encodes destination](../../GLOSSARY.md#bolt11-encodes-destination)) | **HIGH** | inherent to melt; accepted and bounded: defund is the only op that names our node to the mint, so keep defunds infrequent, amount-randomized, window-randomized (§6 T4) |
| check state (`/v1/checkstate`) | AI | queried Y values = which proofs this caller cares about, cadence, IP | MED | links the *requester* to specific tokens, not tokens to issuance; petitioner-side cadence guidance only (§6 T5 - we do not control the AI environment) |
| keys / keysets / info (`/v1/keys`, `/v1/keysets`, `/v1/info`) | both | client software fingerprint, polling cadence | LOW | static discovery; cache responses |

### 5.2 What the AI newly sees or holds

| Surface | Reveals | Sev | Treatment |
|---|---|---|---|
| the float itself | exact AI-held value, denominations, full token contents | by design | **scale cloaking and banding do NOT apply to the float** - a bearer instrument in hand is precisely countable, and pretending otherwise is theater. The float is petty cash (§8); the operator's wallet magnitudes stay cloaked because nothing about them crosses with a funding event |
| `fund_ecash` request/result | the funded amount (the AI chose it), success/failure | LOW | amount knowledge is the AI's own; the allowance cap (§8) bounds the total; reveals nothing about operator wallet size |
| the mint URL and keysets | which mint the operator chose; keyset epochs | MED | inherent (tokens embed the URL; the AI must reach the API). Mitigated by mint choice: busy, public, unaffiliated (§2) - the URL then names a crowd, not the operator |
| the mint channel itself | an operator-sanctioned external comms path from the AI environment: amounts, timing, checkstate queries can encode data outward; token strings carry value to anyone | **HIGH** | the deployment decision (§1). Value exfiltration is bounded by the allowance; data exfiltration over the channel is NOT bounded by spacer and must be acceptable (or separately constrained) in the AI environment's own comms policy before enabling ecash mode |

## 6. Timing: mint-correlation channels

The channels that would let the mint stitch its two views together (arbiter-side LN events, AI-side token events). Numbered in the style of docs 01/03 §4.

| # | Channel | What the mint can infer | Mitigation |
|---|---|---|---|
| T1 | quote -> pay -> issue micro-timing | sub-second gaps fingerprint arbiter automation; a fixed sequence cadence marks all our funding events as one caller | randomized intra-execution gaps between the three steps, inside the action-delay window (landed as `timing.mint_gap_s()`: test mode 0.5-2 s, clearing the sub-second signature while keeping the exit loop fast; production raises `NotImplementedError` pending §10.1) |
| T2 | issuance -> first-swap gap | the handoff token's first swap (the AI claiming its float) trails our LN funding by a recognizable interval; constant gap = the mint links the AI's IP/session to the funding node | the [result delay](../../GLOSSARY.md#result-delay) already randomizes exactly this gap; its window must satisfy the §7 adequacy test against observed mint traffic |
| T3 | amount correlation | a 47,300-sat issuance followed by 47,300 sats of swaps links them regardless of blinding; unique totals are near-fingerprints | randomize funding amounts (non-round, varied); split or batch fundings so issuance totals do not map 1:1 to float top-ups; denominations are powers of 2 by protocol, which helps only if totals vary |
| T4 | defund profile | swap-then-melt pairs terminating at our node pubkey accumulate into a float-lifecycle profile: how much, how often, when | defund infrequently, in randomized amounts, on randomized windows; accept that defund is the profile-richest op (§5.1) and prefer letting the float run down by spending |
| T5 | AI-side cadence | metronomic checkstate or swap polling fingerprints the AI client as automation and sharpens T2 | guidance only (the petitioner wallet can jitter its own calls; the AI environment is outside our control) - documented, not enforced |
| T6 | keyset-epoch partitioning | the anonymity set for every correlation above is the *other traffic in the same keyset epoch*; a quiet mint or a fresh epoch shrinks it toward 1 | mint choice (§2); the adequacy floor (§7) must key on observed mint activity, and §10.1 flags that this activity is not publicly observable |

The arbiter <-> mint link rides the same execution machinery as the daemons but is **not** covered by the internal-link exemption: T1-T4 are mint-facing mitigations on the trusted side's *outbound* behavior, which is precisely where the exemption ends (doc 03 §1 covers the arbiter <-> local bitcoind / LND link only).

## 7. The delay-scaling principle (standing rule, codified)

> **Delays must be adequate for the anonymity-set size and the nuances of the particular privacy concern. Privacy adequacy is the floor; UX never overrides it. The UX bet: delay tolerance tracks rail speed, and the bet pays because the adequate floor is naturally lower on faster rails - the anonymity set turns over faster.**

Bitcoin is inherently slow, so users tolerate large delays - and need them, because the observable surface (global chain) turns over in 10-minute blocks. Lightning is fast, tolerates less - and needs less, because payments are not globally visible and the relevant traffic turns over in seconds. eCash is fast - and a busy mint's swap traffic turns over in seconds-to-minutes. The principle says this alignment is a design input, not a coincidence to hope for: **each op's window is sized to the anonymity set of the surfaces it touches, and the floor binds to the slowest-turnover surface among them.**

| Rail / op class | Observable surface | Anonymity-set source | Production floor |
|---|---|---|---|
| onchain (`manage_bitcoin`) | global chain + mempool | similar txs in mempool/blocks | **~12 h** (unchanged; docs 01/03/05) |
| lightning (`manage_lightning`) | recipient + route nodes | LN payment flow | **~1 h target** (proposed; tuning open, §10.2) |
| ecash boundary (`fund_ecash`, `defund_ecash`) | the mint AND the LN legs | mint mint/melt/swap traffic AND LN flow | **the LN floor governs** (slowest surface touched); mint-leg micro-timing additionally jittered (§6 T1) |
| AI-direct token ops | the mint only | mint traffic | **none** - not arbiter-mediated; the principle's terminal rung |

Consequences. The ~12 h floor in the [action delay](../../GLOSSARY.md#action-delay) / [result delay](../../GLOSSARY.md#result-delay) glossary entries is the **onchain rail's** floor, not a universal constant; this doc is the authority for the per-rail split, and the glossary entries point here. The dynamic-window calculation (doc 05 §7, still open) becomes per-rail: same mechanism, per-rail activity sources and floors. Test mode is unchanged in mechanics (`SPACER_TIMING_MODE=test`, 5-15 s windows) and production timing remains `NotImplementedError`-gated until the dynamic window lands - the safe failure mode carries over to every rail.

What the principle forbids: shrinking a window below adequacy because the rail is fast and users are impatient (UX never overrides), and the inverse laziness of using the onchain floor everywhere because it is safely large (that destroys the extension's reason to exist and trains operators to bypass the gateway).

## 8. Allowance and blast radius

The float replaces per-action control with a pre-committed loss bound. The bound must therefore be hard.

- **`ecash_allowance_sats`** (arbiter config, console-edited like the registry and standing approvals; landed as `config/ecash.yaml`): hard cap on **outstanding float** = Σ funded - Σ successfully defunded. `fund_ecash` refuses when `outstanding + requested > allowance`, audit-logged `decision_refuse_allowance` (with requested / outstanding / allowance in the payload), uniform refusal on the wire. The check precedes standing approvals: **a HITL approval cannot exceed the allowance** - raising it is a config edit at the console, never an approval click. This is deliberate anti-fatigue design: the operator's tired "approve" cannot widen the blast radius. The exit loop proves the ordering, not just the refusal: the over-allowance variant stages a *matching* approval rule and asserts (via forbidden audit events) that the refusal fires without consulting it.
- **Fail-safe default: the arbiter ships with no `ecash.yaml`.** A missing file - or an unreadable, unparseable, or negative value, each audit-logged for operator triage - reads as allowance 0, which refuses every fund. The float cannot exist until the operator explicitly writes its bound.
- **Optional rate cap** (`ecash_funding_rate`): max Σ funded per rolling window, against slow-bleed patterns (spend float to an accomplice, refund, repeat - each step individually approved). The outstanding cap bounds exposure at an instant; the rate cap bounds it over time. Deferred at build time to land with the executor: it needs real funding traffic to tune against, and the outstanding cap is the hard bound.
- **Blast radius = allowance.** Worst-case AI behavior (burn, exfiltrate, gamble) and worst-case mint behavior (rug, freeze) both cost at most the outstanding float. Size it like petty cash: an amount whose *total* loss is acceptable as a cost of the autonomy experiment.
- **What the ledger can and cannot see.** `outstanding` is an upper bound on operator-funded value still in AI hands, maintained entirely from gateway-visible events. AI-direct spends are invisible (by design); tokens *received* from third parties are invisible AND unbounded - the allowance bounds **operator loss**, not AI wealth. A total-holdings bound is unobtainable from a bearer instrument; this doc declines to pretend otherwise.
- **No clawback.** Bearer means bearer: the operator cannot revoke the float, only stop funding it and let spending or a defund run it down. The mint cannot help (blinding means it does not know whose proofs are whose). P2PK-locked handoff tokens with a timelocked operator refund path could buy a sweep capability at real complexity cost - parked as §10.3.
- **Audit.** Every fund/defund records amount, outstanding before/after, and decision path; the float trend is operator-readable at the console even though individual AI spends are not. Landed as the append-only `ecash_ledger` table (executor-written on successful executions; audit events `ecash_ledger_fund` / `ecash_ledger_defund`). One build refinement: **`outstanding` floors at 0** - a defund surplus (e.g. third-party-received tokens defunded through us) must not go negative and thereby widen the funding headroom past the allowance.

## 9. Mode gating and op routing

- **`SPACER_MODE=ecash`** enables the full ladder: onchain + lightning + ecash surfaces. The extension cannot exist without LND (fund/defund are LN legs), so ecash mode implies the Lightning extension exactly as lightning mode implies bitcoind.
- **`full` stays frozen at its 2026-06 meaning** (onchain + lightning) as a legacy alias of `lightning`. Rationale: an extension that moves bearer value out of gateway control must never switch on without the operator typing its name. Letting `full` auto-grow would silently arm ecash on existing `full` deployments at upgrade - the exact opposite of the explicit-opt-in stance the mode gate encodes.
- **Fail-safe parse unchanged:** anything outside `{lightning, full, ecash}` - including typos and unset - runs onchain.
- **Ops:** `fund_ecash` / `defund_ecash` join the write set in ecash mode only (`_ECASH_WRITE_OPS`, mirroring `_ADVANCED_WRITE_OPS`). No new read ops: the AI counts its own float locally, and `query_balance` keeps its lightning-mode behavior (the LND wallet) in ecash mode. The extension-gated refusal generalizes: in onchain mode, LN and eCash ops refuse `decision_refuse_mode` (reason `advanced_extension_disabled`, the op field disambiguates which extension); in lightning mode, eCash ops refuse the same way. No HITL park - the operator already decided by choosing the mode.
- **Lazy import, extended:** `arbiter/src/ecash.py` (the nutshell-CLI wrapper: one `_run` helper, argv list, no shell) is imported only on ecash-mode dispatch, mirroring `_lnd()`. An onchain or lightning deployment never loads it and carries no nutshell dependency at runtime. Build detail: the first import also registers and applies the `ecash_ledger` schema (an idempotent re-migrate), so the boot path stays ignorant of the module.
- **petcli:** the eCash commands live under the advanced namespace as `advanced ecash {fund, defund, balance, send, receive, info}`. `fund` / `defund` speak to the arbiter; the rest are local wallet / direct mint operations - as built, they shell out to a petitioner-side `cashu` CLI (`PETCLI_CASHU_BIN`; PATH resolution is acceptable there, since the petitioner holds no operator secrets) and wrap the verbatim stdout/stderr/exit code in a `_petcli_local` envelope, distinguishing a local wallet outcome from anything the arbiter said (the sibling of `_petcli_transport_error`, doc 05 §5.1). The namespace stays always-visible for discovery; the gate lives arbiter-side, as today (doc 05 §5.1).
- **Exit loop (landed with sp-2hwco4.2):** 42 manifest variants plus the no-lnd-import and no-ecash-import gates, all green. The eCash additions cover both custody halves: mode-gate refusals for `fund_ecash` / `defund_ecash` against onchain AND lightning arbiters; the allowance gate (missing-config default-deny, and the over-allowance refusal proven to fire before approvals, §8); both standing-approvals branches for fund and defund (`destination: mint`); ladder-regression variants (`query/balance/ecash-lnd-wallet`, `advanced/channels/ecash-mode`) holding the Lightning surface exactly as lightning mode has it under ecash mode; and the local wallet commands against a fake petitioner-side `cashu` (`$CASHU_SCENARIO`, mirroring `$BITCOIN_CLI_SCENARIO` / `$LNCLI_SCENARIO`), plus a deterministic missing-binary error variant. Deliberately, **no fake exists for the arbiter-side wrapper**: no manifest variant can reach it (eCash writes stop at the gates or the dispatch stub), and leaving `CASHU_BIN` / `CASHU_MINT_URL` unset means an unexpected arbiter-side mint call errors loudly instead of being absorbed. Onchain and lightning artifacts stayed byte-identical modulo timestamps/ports. As of sp-uwa0v0 the happy-path fund/defund (and manage_bitcoin/manage_lightning) ride the landed timing-layer executor: the four `allowed-by-standing-approval` variants now assert the timing-layer acknowledgment (`status: received` + handle), not the old `not_implemented` stub, and a `--live` mode drives all four write ops through the real executor against Node A + `cashu.mutinynet.com` (doc 05 §11). The live nutshell mint selection and round-trip flow are `08--2026-06-18-0629-ecash-live-test-mint.md`, with the live results in that doc's findings companion.

## 10. Open questions

1. **Mint-activity source for the dynamic window.** Chain activity is publicly observable; a mint's swap volume is not (`/v1/info` exposes no traffic data). Candidates: conservative static randomized windows until measured; inference from keyset-rotation cadence; operator-side probing. Gates production timing for the ecash rail, parallel to doc 05 §7.
2. **The lightning-rail floor.** ~1 h is proposed on turnover reasoning, not measurement. Needs validation against real LN/mint traffic before production windows exist anywhere.
3. **P2PK-locked handoff tokens (NUT-11) with timelocked operator refund.** Was deferred to reconcile time; **resolved at reconcile (sp-2hwco4.3): not adopted.** Plain bearer handoff tokens stand. The allowance already prices in total loss of the float (§8: size it like petty cash), so a sweep-stale-float capability recovers value the operator has by definition already written off - it does not buy down any unaccepted risk. The costs named here all stand: key management, a bigger arbiter surface against the §2.1 minimalism discipline, and nutshell CLI support uncertainty (still unverified - the build never reached a live mint). Revisit on the same trigger as the §2 Fedimint rejection: float sizes outgrowing the petty-cash framing.
4. **Fee accounting.** Mint input fees, LN routing fees, and the melt fee reserve make funded != received and defunded != credited. Proposal: the outstanding ledger tracks gross token value; fees are operator cost, audit-logged per op. Build status: the ledger records gross amounts per the proposal; the live round-trip (sp-uwa0v0) measured the fees - funded != credited is dominated by the deliberate melt reserve (~2%), with ~1-2 sat genuine LN/mint fee, and this mint charged no mint/melt input fee (findings companion §2).
